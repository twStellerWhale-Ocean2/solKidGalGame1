# 專用性規範-管理網站

# I. 主旨目的

本規範適用於管理網站、Dashboard、MIS、作戰總覽、企業資料操作介面、AI 平台管理介面與所有需長時間操作的後台系統。

管理網站必須是可操作產品，不得做成 landing page、靜態展示頁、單頁塞滿 CRUD 的臨時介面，或以角色下拉選單假裝登入與權限。

# III. 內容程序

## A. Layout 與 Navigation

Desktop 為主要操作環境，主架構固定為：

```text
Top App Bar
Left Navigation Drawer
Main Content Area
Optional Right Detail Panel
```

禁止主導航放右側、多重主導航、漂浮型主要選單或滿版自由排版。

Navigation Drawer 固定在左側，寬度為 280dp，收合後為 80dp。內容順序為：

```text
Logo
Main Navigation
Sub Navigation
Spacer
User Information
```

Top App Bar 固定高度 64dp。左側放 Menu Button 與 Breadcrumb，中間放 Page Title，右側放 Notification、Search、User Profile。

Breadcrumb 固定存在於 Page Title 上方，格式為：

```text
Home > Module > Function
```

## B. Dashboard 與 KPI

Dashboard 採 Card-based、Data-first、Scan-friendly。Dashboard Card 必須包含：

- Title
- Main Value
- Delta
- Time Reference

KPI 色彩語意：

```text
Success: Green
Warning: Orange
Critical: Red
Info: Cyan
```

Dashboard 第一屏應優先呈現作戰總覽、主要 KPI、警示、近期活動與待處理事項，不得用行銷式 hero 或純說明文字取代。

## C. Button 與 Action 位置

Primary Button 使用 Filled 樣式並作為高亮主要操作，每個區域最多一個。Secondary Button 使用 Outlined。Danger Button 使用 Red，且必須搭配 Confirm。

Button 高度固定為 40dp。

CRUD Toolbar 操作固定在右上角。Dialog Action 固定在右下角，順序為：

```text
Cancel | Confirm
```

Danger 操作順序為：

```text
Cancel | Delete
```

Read / Edit 使用 Material-style icon button，不使用文字按鈕；常用 icon 為 `visibility` / `edit`。Create、Save、Refresh、Search、Filter、Export 亦優先使用 icon 搭配必要 label。

不可用 action 必須保留 disabled icon button，並以 tooltip 或等價可視方式說明不可用原因；不得讓按鈕直接消失造成權限或狀態不明。

## D. CRUD 操作規範

Create 優先使用 Dialog 或 Right Drawer，避免 Full Page Create；若表單複雜到需分段，才使用獨立 Create 頁。

Read 優先使用 Table、Card、Detail Panel 或 Right Drawer。Update 優先使用 Inline Edit、Dialog 或 Right Drawer。

Delete 不採各列自行提供 Delete 作為主要模式；資料清單必須在最左側提供 multi-select checkbox，使用者可勾選多筆後從 Toolbar 執行 bulk delete 或 bulk disable。

Delete 必須使用 Confirm Dialog，且必須顯示刪除目標名稱、筆數與不可復原或停用影響。若資料因權限、狀態、關聯資料或稽核要求不可刪除，仍顯示 disabled bulk action 或 row action，並用 tooltip 說明原因。

CRUD 不是畫面樣式而已，必須實際完成資料狀態轉移：建立後可在 List 看到，Read 可讀到同一筆，Edit 後 List/Read 反映新值，Delete/Disable 後狀態可被查證。

## E. Table 規範

表格採 Dense Enterprise Style，Row Height 為 52dp。

Table 必須支援：

- Sorting
- Filtering
- Pagination
- Search
- Multi-select Checkbox

Multi-select Checkbox 固定在最左側。Action Column 固定在最右側。Row action 應維持一致順序，例如：

```text
Read icon | Edit icon | More icon
```

Delete / Disable 以 bulk action 為主，避免每列塞滿危險操作。若需要列級危險操作，必須放在 More menu 並二次確認。

## F. Form 規範

Label 固定放在 Input 上方。Required 使用紅色 `*`。Validation 錯誤訊息放在 Input 下方，以紅色顯示，不跳 Toast。

Form Width 禁止超寬表單與滿版 Input，建議控制在 720px 以下。

表單需要支援：

- Required / Optional 區分
- Inline validation
- Save / Cancel 明確動作
- Dirty state 提醒
- 成功與失敗狀態回饋

## G. 登入與權限呈現

若 story/case 涉及登入或權限，產品介面應採業界常見登入流程、個人資訊與權限狀態呈現；不得用頁面上的角色下拉選單作為正式 runtime 的身份切換方式。

測試或 demo 需要切換身份時，可以使用測試帳號、測試登入流程或明確標示的測試工具，但不得混入正式產品主流程並冒充實際權限設計。

## H. 狀態回饋與錯誤處理

所有主要操作都必須有可見狀態：loading、success、failure、empty、disabled、permission denied、validation error、degraded。

錯誤訊息須指出發生位置、原因與下一步；不得只顯示 `failed`、`error` 或無回饋。

## I. AI 功能呈現

AI 問詢、AI 建議、AI 警示或 AI 分析區塊應清楚呈現資料來源邊界、可信程度、不可回答或服務降級狀態。

AI 成功路徑不得以固定文字、前端本地規則或靜態 FAQ 冒充。若為 demo mode、mock、stub 或降級回覆，必須明示。

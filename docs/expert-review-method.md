# 專家會審（Expert Peer Review）— 逐頁 UX／美術審查方法

本文件定義 [solKidGalGame方案] 的「專家會審」測試手法（issue #103 試作版）。以「幼兒英語學習軟體／兒童 UI·UX／視覺設計」業界專家視角，**逐頁**檢視每個畫面的需求滿足與美術一致性，產出可回修的問題清單，並納入 [docs/test-summary.html]。

## 定位與沿革

- **承繼**既有 `GATE.md §6「專家評價」`精神（專家視角＋每頁 ≥10 項發現＋重大回修＋報告，產出如 `docs/expert-review-issue93.md`），更新為**逐頁 UI·UX＋美術一致性**、且**納入 test-summary** 的新版做法。
- 本版為 **issue #103 試作**：先在本 repo 驗證做法可行，**暫不**改全域方法論 SKILL（`FORMAT.md`／`GLOSSARY.md`／`docLint.ps1`／`SKELETON.md`），亦**不**進入 [docs/design.md] ＜III. 測試規格＞——因該章受 docLint 限制為封閉測試節 `A–F`、6 類測試分類為封閉集，新增測試類別屬全域格式變更。驗證可行後，另於 skill repo 立 Issue 正式形式化（屆時一併界定與 GATE §6 之承繼/取代關係）。
- **棒次分工**：審查規範（本文件＋頁面清單）由 2plan 釘定；逐頁截圖、四項分析、≥10 問題分級、`務必要修` 回修與 test-summary 產製，由 3dev 執行。

## 測試單元＝頁面/畫面

「頁面」指使用者實際看到的每一個畫面狀態（含主視圖與覆蓋層 modal）。本專案為純前端靜態 web（`techStackStaticWeb`），其畫面取自 [index.html]，正規清單如下（3dev 以 DOM 錨點將 app 導至該狀態截圖）：

| 代號 | 頁面/畫面 | DOM 錨點 | 使用者主要企圖（功能＋美觀便利） |
|---|---|---|---|
| ER-P01 | 帳號選擇 Account Select | `#accountSelect` | 選/新增/刪除本機帳號；快速辨識是誰的進度 |
| ER-P02 | 選角與命名 Character Select | `#characterSelect` | 挑公主外觀、命名、開始；即時預覽外觀 |
| ER-P03 | 公主狀態側欄 Side Panel | `aside.side-panel` | 隨時看 coins／遊玩時間／穿搭；開系統選單、換公主 |
| ER-P04 | 城堡地圖 Castle Map | `#homeView` | 在城堡內選地點進入；看自己位置 |
| ER-P05 | 世界地圖 World Map | `#worldView` | 選地區（Castle/Urban/Rural/Wild）旅行 |
| ER-P06 | 地區地圖 Area Map | `#mapView` | 在地區內選地點；導航、拖曳、標記辨識 |
| ER-P07 | ADV 對話／答題 Scene | `#advModal`（`#advLine`/`#choiceList`/`#advFeedback`） | 聽情境句、選英文答案、得回饋；撥放英/中、求助 |
| ER-P08 | Help 提示 | `#helpButton` 觸發之提示 | 看不懂時取得提示、理解題意 |
| ER-P09 | 商店 Shop | `#advModal #shopArea`（`#advShopGrid`） | 用 coins 買外觀；看價格、分類、可否購買 |
| ER-P10 | 衣櫃／換裝 Wardrobe | adv 面板內 detail panel（item-panel） | 試穿/穿戴/退款；看穿搭效果 |

## 逐頁審查程序（每頁依序）

1. **截圖**：將 app 導至該頁代表狀態，截取現有畫面（行動直向為主、桌機為輔），存入 test-summary 對應頁區塊。
2. **企圖與需求分析**：以該頁使用者（幼兒玩家／陪伴家長）視角，列出主要企圖與需求——功能面（主要任務）＋美觀便利性（次要期待，如查詢、跳轉快速選單、回上層、回饋可見性）。
3. **需求滿足分析**：逐項判斷上述需求於現況是否被滿足（滿足／部分／未滿足＋說明），納入 test-summary。
4. **美術常規分析**：對照業界兒童 app 常規（見〈業界常規對標〉），檢查版面、色彩、字級、間距、對比、觸控目標尺寸、元件一致性等是否符合常規——目標是「不搞特例」而非追求新奇，納入 test-summary。
5. **≥10 問題並分級**：每頁至少列 10 個問題，逐一標 `務必要修`／`後續辦理`／`可以接受`，納入 test-summary。
6. **`務必要修` 回修**：所有 `務必要修` 問題須於本棒（3dev）修復完成，test-summary 顯示修復結果。

## 問題分級

- **`務必要修`**：阻礙幼兒完成核心任務、明顯違反兒童可用性/無障礙常規、或明顯美術錯誤（破版、元件重疊、不可讀、觸控目標過小）。**本批次必修**。
- **`後續辦理`**：影響體驗但不阻礙任務；列入 backlog，非本批次必修。
- **`可以接受`**：刻意設計取捨或低影響；記錄即可、不必處置。

## test-summary 逐頁區塊範本

test-summary 以每頁為一個測試單元，每頁含一區塊：

```md
### [ER-P0N] 頁面名（DOM 錨點）
- 截圖：<行動直向截圖>（必要時附桌機）
- 使用者企圖與需求：功能 1)… 2)…；美觀便利 a)… b)…
- 需求滿足：每項需求 → 滿足／部分／未滿足 ＋說明
- 美術常規：符合項／偏離項（對標業界）
- 問題清單（≥10，分級）：

  | # | 問題 | 分級 | 修復結果 |
  |---|---|---|---|
  | 1 | … | 務必要修 | 已修：… |
  | 2 | … | 後續辦理 | — |
  | …（至少 10 列）… |
```

## 業界常規對標

沿用 [README.md]〈參考案例〉作為兒童 app 常規基準，作為「美術是否符合業界常規」之判準：

- **Khan Academy Kids**：角色陪伴、短任務、學習足跡。
- **Duolingo ABC**：短回合、低挫折、即時回饋。
- **Lingokids**：兒童可自行操作之導覽。
- **Toca Boca World**：換裝、自我表達、角色扮演。

常規檢查重點：大觸控目標、清楚即時回饋、低文字依賴（幼兒未識字）、一致導覽（場景第一層 `Leave`、細項面板 `Back`）、可讀字級與足夠對比、行動直向優先。

## 範圍與已知考量

- 本試作涵蓋上表**全部 15 頁**（依 issue #103「包含所有頁面」）。
- 量級：15 頁 × ≥10 問題 ≈ ≥150 項；3dev 可分批執行，但 `務必要修` 須全數修復。
- 「每次 Issue 應覆蓋全頁或僅異動頁」之長期範圍規則，與 GATE §6 之承繼/取代關係，**留待形式化 Issue 決定**；本試作目的為建立全頁基準並驗證方法可行性。

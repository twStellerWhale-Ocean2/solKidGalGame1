# 專用性規範-說明網站

# I. 主旨目的

本規範適用於產品說明網站、方案介紹頁、靜態 HTML/CSS/JS 展示網站、文件導覽頁、GitHub organization/profile README 與對外介紹頁。

說明網站與管理網站不同；說明網站可以偏介紹性、敘事性與行銷性，但仍必須清楚呈現產品、方案、對象、能力、操作入口與可信資訊，不得用空泛 hero、裝飾圖或沒有內容的靜態頁充數。

# II. 參考準備

* [ ] 先讀取 `通用性規範.md`。
* [ ] 盤點說明網站需要呈現的產品名稱、目標使用者、主要價值、功能清單、流程、截圖、FAQ、文件連結、操作入口與聯絡方式。
* [ ] 若成果是 GitHub README 或 Markdown 文件導覽，盤點首頁、子頁、資源頁、返回路徑、外部連結與每個子頁的完成狀態。
* [ ] 若說明網站會引導到管理網站或系統入口，確認連結、權限、登入與環境說明。

# III. 內容程序

## A. 首屏與品牌呈現

首屏必須讓使用者立刻知道這是什麼產品/方案、服務誰、解決什麼問題，以及下一步可做什麼。

Landing hero 可以使用真實產品截圖、生成式概念圖、流程圖或情境圖，但不得用純漸層、空泛抽象背景或不相關 stock 圖取代產品資訊。

H1 應是品牌、產品、方案名稱或明確類別；價值主張放在 supporting copy，不要把抽象口號當成唯一標題。

首屏需保留下一段內容的可見提示，避免整屏只有 hero。

## B. 網站結構

常見結構：

```text
首頁
├── 方案總覽
├── 核心能力
├── 使用情境
├── 系統架構或流程
├── 截圖/展示
├── 安全與權限
├── 文件/開發者資源
├── FAQ
└── 進入系統 / 聯絡 / 下載
```

說明網站可採 section scroll，但導覽列必須清楚，重要 CTA 必須可被找到。

若是 GitHub organization/profile README、文件首頁或 Markdown 型說明網站，常見結構：

```text
首頁 README
├── 品牌/組織定位
├── 主軸或產品線總覽
├── 子頁導覽表
├── 依角色的閱讀順序
├── 資源與聯絡入口
└── 子頁
    ├── 定位
    ├── 適用情境
    ├── 核心工作或能力
    ├── 與其他頁面的關係
    └── 下一步 / 返回首頁
```

Markdown 導覽表應讓主要項目名稱本身成為連結，例如 `[英仙座系統](./pages/perseus-system.md)`；不得把所有項目都放在非語意的「閱讀子頁」「入口」「更多」欄位中，否則掃讀時無法直接判斷可點擊目標。

首頁應保留摘要與導覽，不應把所有子頁細節塞回同一頁。子頁必須能獨立閱讀，不得只有一段簡介或尚未完成的 placeholder。

## C. 視覺風格

說明網站可比管理網站更具敘事性，但仍維持 Material Design 3、AI startup、乾淨專業與高可讀性。

允許：

- 產品截圖與重點標註。
- 情境圖、流程圖、架構圖。
- 適度動態與過場。
- 清楚的 feature section。
- 內容卡片與比較表。

禁止：

- 只有 hero 沒有實質內容。
- 大量浮誇漸層、霓虹、玻璃擬態或裝飾粒子。
- 用 marketing copy 取代操作資訊。
- 將管理介面功能塞成單頁 CRUD demo。

## D. 內容要求

每個主要功能區塊至少包含：

- 功能名稱。
- 解決的問題。
- 對應使用者或角色。
- 可看到或可操作的成果。
- 與其他系統或資料的關聯。

若功能涉及 AI、LINE、IdP、外部 API 或跨系統服務，必須說明資料來源邊界、使用者可期待的效果與服務不可用時的處理方式。

Markdown 型說明網站的子頁至少包含：

- 頁面標題與一句清楚定位。
- 適用情境或目標對象。
- 核心能力、工作項目或交付內容。
- 與其他子頁、系統或資源的關係。
- 下一步、相關頁面或返回首頁連結。

應避免以下內容問題：

- 「...」「待補」「更多資訊」等未完成字樣。
- 只把舊資料搬過來，仍保留「舊公開首頁」這類遷移痕跡。
- 每個子頁格式不一致，導致讀者無法預期下一段內容。
- 外部連結與內部連結混在同一欄位但沒有說明。
- 只有抽象願景，沒有對應使用者、情境或可交付成果。

## E. 響應式與可讀性

說明網站應支援 desktop、tablet、mobile。Mobile 可重排內容，但不得讓按鈕文字、卡片內容或導覽互相遮蔽。

文字不可依 viewport width 無限制縮放；用明確的字級階層與容器寬度控制可讀性。

## F. 靜態資源實作

若以靜態 HTML/CSS/JS 實作，仍須符合：

- 導覽可用。
- CTA 可點擊。
- 截圖或媒體可載入。
- Section 錨點正確。
- Console 無主要錯誤。
- 檔案路徑與部署目標一致。

# IV. 備註紀錄

* 本規範補足原視覺規範未細分的說明網站情境，避免將管理網站規範誤套成行銷頁，或將說明網站做成空泛靜態頁。
